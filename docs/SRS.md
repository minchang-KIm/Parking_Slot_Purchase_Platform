# 소프트웨어 요구사항 명세서 (SRS)
# 주차 거래 플랫폼

**버전:** 1.0
**작성일:** 2025년 11월 17일
**팀:** 1조

---

## 1. 서론

### 1.1 목적
본 문서는 개인간 주차 공간 거래 플랫폼의 기능적 및 비기능적 요구사항을 정의합니다. 이 문서는 개발팀, 테스트팀, 프로젝트 관리자를 위한 참조 자료로 사용됩니다.

### 1.2 범위
주차 거래 플랫폼은 주차공간을 제공하는 사용자와 주차공간을 필요로 하는 사용자를 연결하는 웹 기반 플랫폼입니다.

### 1.3 정의 및 약어
- **사용자 (User)**: 주차공간을 예약하는 일반 사용자
- **제공자 (Provider)**: 주차공간을 등록하고 제공하는 사용자
- **관리자 (Admin)**: 플랫폼 관리 권한을 가진 사용자
- **API**: Application Programming Interface
- **JWT**: JSON Web Token

## 2. 전체 설명

### 2.1 제품 관점
주차 거래 플랫폼은 다음의 주요 서브시스템으로 구성됩니다:
- 사용자 인증 및 권한 관리 시스템
- 주차공간 등록 및 검색 시스템
- 예약 및 결제 시스템
- 리뷰 및 평점 시스템
- 관리자 대시보드

### 2.2 제품 기능
1. **회원 관리**: 회원가입, 로그인, 프로필 관리
2. **주차공간 관리**: 등록, 수정, 삭제, 검색
3. **예약 시스템**: 예약 생성, 취소, 확인
4. **결제 시스템**: 결제 처리, 환불
5. **리뷰 시스템**: 리뷰 작성, 조회, 관리
6. **관리자 기능**: 사용자 관리, 통계 조회

### 2.3 사용자 특성
- **일반 사용자**: 주차공간 검색 및 예약
- **공간 제공자**: 주차공간 등록 및 관리
- **관리자**: 플랫폼 전체 관리

### 2.4 제약사항
- 웹 브라우저 환경에서 동작
- MongoDB 데이터베이스 사용
- RESTful API 아키텍처
- JWT 기반 인증

## 3. 기능 요구사항

### 3.1 사용자 관리
**FR-1.1** 시스템은 사용자가 이름, 이메일, 비밀번호, 전화번호를 입력하여 회원가입할 수 있어야 한다.
**FR-1.2** 시스템은 이메일과 비밀번호를 통한 로그인 기능을 제공해야 한다.
**FR-1.3** 시스템은 사용자 프로필 조회 및 수정 기능을 제공해야 한다.
**FR-1.4** 시스템은 비밀번호 변경 기능을 제공해야 한다.

### 3.2 주차공간 관리
**FR-2.1** 제공자는 주차공간을 등록할 수 있어야 한다. (제목, 설명, 주소, 위치, 가격, 타입)
**FR-2.2** 시스템은 위치 기반 주차공간 검색 기능을 제공해야 한다.
**FR-2.3** 시스템은 가격, 타입, 특징으로 주차공간을 필터링할 수 있어야 한다.
**FR-2.4** 제공자는 자신의 주차공간을 수정 및 삭제할 수 있어야 한다.
**FR-2.5** 시스템은 주차공간 상세 정보를 제공해야 한다.

### 3.3 예약 시스템
**FR-3.1** 사용자는 주차공간을 예약할 수 있어야 한다.
**FR-3.2** 시스템은 예약 시간 중복을 확인해야 한다.
**FR-3.3** 시스템은 예약 가격을 자동 계산해야 한다.
**FR-3.4** 사용자는 예약을 취소할 수 있어야 한다.
**FR-3.5** 제공자는 예약을 확인할 수 있어야 한다.
**FR-3.6** 시스템은 예약 내역을 조회할 수 있어야 한다.

### 3.4 결제 시스템
**FR-4.1** 시스템은 다양한 결제 수단을 지원해야 한다. (카카오페이, 토스)
**FR-4.2** 시스템은 결제 내역을 저장하고 조회할 수 있어야 한다.
**FR-4.3** 시스템은 환불 기능을 제공해야 한다.
**FR-4.4** 시스템은 결제 실패 시 적절한 에러 메시지를 표시해야 한다.

### 3.5 리뷰 시스템
**FR-5.1** 사용자는 완료된 예약에 대해 리뷰를 작성할 수 있어야 한다.
**FR-5.2** 리뷰는 1~5점 평점과 텍스트 코멘트를 포함해야 한다.
**FR-5.3** 시스템은 주차공간별 평균 평점을 계산하고 표시해야 한다.
**FR-5.4** 제공자는 리뷰에 답변할 수 있어야 한다.
**FR-5.5** 사용자는 리뷰를 "도움됨"으로 표시할 수 있어야 한다.

### 3.6 관리자 기능
**FR-6.1** 관리자는 대시보드에서 플랫폼 통계를 조회할 수 있어야 한다.
**FR-6.2** 관리자는 사용자를 관리할 수 있어야 한다.
**FR-6.3** 관리자는 주차공간을 승인/거부할 수 있어야 한다.
**FR-6.4** 관리자는 모든 예약 및 결제 내역을 조회할 수 있어야 한다.

## 4. 비기능 요구사항

### 4.1 성능 요구사항
**NFR-1.1** 시스템은 일반 API 요청에 대해 평균 2초 이내에 응답해야 한다.
**NFR-1.2** 시스템은 동시에 1000명의 사용자를 지원해야 한다.

### 4.2 보안 요구사항
**NFR-2.1** 모든 비밀번호는 암호화되어 저장되어야 한다.
**NFR-2.2** API 통신은 HTTPS를 사용해야 한다.
**NFR-2.3** 인증은 JWT 토큰 기반으로 구현되어야 한다.

### 4.3 사용성 요구사항
**NFR-3.1** 사용자 인터페이스는 직관적이고 사용하기 쉬워야 한다.
**NFR-3.2** 시스템은 반응형 디자인을 지원하여 모바일 환경에서도 사용 가능해야 한다.

### 4.4 신뢰성 요구사항
**NFR-4.1** 시스템 가용성은 99% 이상이어야 한다.
**NFR-4.2** 데이터베이스는 정기적으로 백업되어야 한다.

### 4.5 유지보수성 요구사항
**NFR-5.1** 코드는 명확한 주석과 문서화가 되어야 한다.
**NFR-5.2** 시스템은 모듈화되어 유지보수가 용이해야 한다.
